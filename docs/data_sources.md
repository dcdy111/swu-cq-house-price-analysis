## 旧库来源判断

当前 MySQL 中旧库导入结果：

- `anjuke_legacy`：118,661 条
- `lianjia_legacy`：3,116 条
- `fang`：60 条新采样本

因此旧项目主体更接近安居客采集结果，链家样本量较少。旧库字段完整度较好，但真实性不能直接证明，必须通过质量评分、异常值、重复检查和新采样本对照来降低风险。

## 本系统已经做的清洗与整理

1. 字段标准化：统一 `source/title/link/district/community/total_price/unit_price/area/layout` 等字段。
2. 单位转换：总价统一为“万元”，单价统一为“元/平方米”，面积统一为“平方米”。
3. 户型解析：从 `layout` 中提取 `rooms`、`halls`。
4. 楼层归一：从楼层文本派生 `floor_level = low/mid/high/unknown`。
5. 建成年份与房龄：解析 `build_year`，派生 `house_age`。
6. 去重指纹：按 `source + fingerprint` 去重，fingerprint 不包含价格。
7. 增量快照：首次入库写入 `listing_snapshots`，价格或单价变化时追加快照。
8. 数据质量分：按关键字段缺失、价格异常、面积异常、建成年份异常扣分。
9. 来源分层：旧库和新爬数据使用不同 `source`，避免混在一起无法解释。
10. 异常保留：异常房源不物理删除，后续分析建模可默认过滤。

## 展示建议

学年设计应当展示“数据清洗与整理”过程。建议在报告和系统中展示：

- 原始数据样例 -> 清洗后标准字段样例。
- 来源分布：各平台样本量和区县覆盖。
- 质量报告：缺失率、异常值数量、重复数量、平均质量分。
- 增量证明：重复采集不重复入库，价格变化写入快照。
- 风险说明：旧库作为冷启动基线，新标准采集样本作为主要分析依据。

这部分能体现系统不是简单爬取和展示，而是完成了数据工程治理闭环。

## 部署口径

第一版部署推荐：

```text
Nginx 静态托管前端 dist
Nginx /api 反向代理到 Gunicorn
Gunicorn 运行 Flask API
systemd 保活后端服务
MySQL 8.x 作为唯一运行数据库
```

不建议第一版使用 Docker。原因是课程设计答辩更看重可解释、可维护、可截图验证；`Nginx + Gunicorn + systemd + MySQL` 更容易展示进程状态、数据库状态和接口状态。

爬虫部署策略：

- 服务器端只默认启用房天下的小规模增量任务。
- 安居客和链家作为实验源，需要 Cookie 或浏览器会话时在本地采集后导入。
- 不绕验证码，不做强对抗反爬；遇到风控页记录任务日志。
