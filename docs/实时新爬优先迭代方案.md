# 实时新爬优先迭代方案

更新日期：2026-06-25

## 1. 口径修正

后续课程验收主线改为：

```text
当前系统实时新爬 -> 清洗入 MySQL -> 增量快照 -> Dashboard/模型/Agent 证据
```

旧 SQLite/旧库只作为开发期参考，不作为最终答辩核心数据来源。答辩时如果数据库里还保留旧库数据，也必须按来源分层说明，不能把旧库数据混入“实时新爬 5 万条”的验收口径。

## 2. 当前已经补强的能力

- 支持 `districts=["all"]` 一键创建全区县采集任务。
- 每区最大页数从固定 50 改为配置项 `CRAWL_MAX_PAGES_PER_DISTRICT`，默认 200，设置页可改。
- 支持失败重试次数 `CRAWL_RETRY_TIMES`，默认 2，设置页可改。
- 支持从 `.env` 注入浏览器请求头和 Cookie：
  - `FANG_HEADERS_JSON` / `FANG_HEADERS_RAW` / `FANG_COOKIE`
  - `ANJUKE_HEADERS_JSON` / `ANJUKE_HEADERS_RAW` / `ANJUKE_COOKIE`
  - `LIANJIA_HEADERS_JSON` / `LIANJIA_HEADERS_RAW` / `LIANJIA_COOKIE`
- 页级采集结果增加证据字段：HTTP 状态、最终 URL、耗时、HTML 字节数、重试次数。
- 采集任务结果增加吞吐指标：耗时、页/分钟、条/分钟、失败率。
- 新增只探测不入库脚本：`scripts/crawl_probe.py`。

## 3. Header/Cookie 提供格式

推荐你从浏览器 Network 中复制目标列表页请求。优先给这几类信息：

1. 列表页 URL：例如重庆某区县二手房第一页和第二页。
2. Request Headers：优先 JSON；也可以原始多行 Header。
3. Cookie：只给本地使用，不提交到仓库。
4. 页面是否登录后可访问、是否遇到验证码。
5. HTML 中房源卡片的几个字段样例，或者直接保存页面 HTML。

`.env` 示例：

```text
FANG_HEADERS_JSON={"User-Agent":"...","Referer":"https://cq.esf.fang.com/"}
FANG_COOKIE=key=value; key2=value2

ANJUKE_HEADERS_RAW=User-Agent: ...;;Referer: https://m.anjuke.com/cq/sale/
ANJUKE_COOKIE=key=value; key2=value2

LIANJIA_HEADERS_JSON={"User-Agent":"...","Referer":"https://m.lianjia.com/cq/ershoufang/"}
LIANJIA_COOKIE=key=value; key2=value2
```

注意：不要把真实 Cookie、Token、账号密码写进 README、截图或 Git。

## 4. 探测命令

不入库，只验证访问和解析：

```bash
python scripts/crawl_probe.py --source fang --district 渝中 --page 1 --show-first 3
```

带浏览器 Header/Cookie：

```bash
python scripts/crawl_probe.py ^
  --source anjuke_mobile ^
  --district 渝北 ^
  --page 1 ^
  --headers-json "{\"User-Agent\":\"...\",\"Referer\":\"https://m.anjuke.com/cq/sale/\"}" ^
  --cookie "key=value; key2=value2"
```

成功标准：

```text
ok = true
status_code = 200
listing_count > 0
samples 中 title/link/district/total_price/unit_price/area 至少大部分可解析
```

如果返回验证码、登录页、空列表或跳转页，不硬试；保存输出，调整 Header/Cookie 或换数据源。

## 5. 5 万条新爬路线

推荐三阶段推进：

### 阶段 A：单页探测

每个来源先测：

- 房天下：3 个区县，每区 1 页。
- 安居客移动端：3 个区县，每区 1 页。
- 链家移动端：有 Cookie 后 2 个区县，每区 1 页。

目标：确认哪个来源当前最稳定，哪些字段质量最好。

### 阶段 B：小规模真实入库

建议命令或页面参数：

```json
{
  "source": "fang",
  "districts": ["渝中", "江北", "渝北", "南岸", "沙坪坝", "九龙坡"],
  "max_pages": 5,
  "max_workers": 4,
  "mode": "manual",
  "run_now": true
}
```

目标：形成几百到一两千条新标准样本，验证去重、快照、日志、Dashboard 和模型可用。

### 阶段 C：全区县扩展

配置：

```json
{
  "source": "fang",
  "districts": ["all"],
  "max_pages": 80,
  "max_workers": 4,
  "mode": "manual"
}
```

按结果逐步提高 `max_pages`，不要一上来把 200 页全开。全量目标不是瞬间冲满，而是稳定拿到 5 万条可解释、可追踪、可复采的数据。

## 6. 并发策略

当前建议：

- 默认并发：4。
- 稳定后可试：6。
- 不建议无证据直接上 10+。
- 请求间隔保留 1-3 秒；如果来源稳定，可在局部探测时降低，但不要作为默认答辩策略。
- 遇到验证码/登录/反爬网关，记录日志并停止扩散，不做强对抗。

“极致高效”的正确方向不是盲目加线程，而是：

1. 静态请求优先，避免浏览器自动化。
2. 一页解析多条，减少详情页依赖。
3. 按区县和页码切片并发。
4. 失败页可重试、可追踪、可补采。
5. DB 写入用去重和快照保证重复采集不膨胀。

## 7. Agent 工具调用证据链

答辩时 Agent 必须展示：

1. 用户问题。
2. 工具规划，例如 `query_market_stats`、`get_model_result`、`generate_report`。
3. 工具 input。
4. 工具 output。
5. 最终回答。
6. `agent_tool_calls` 中的记录。
7. 报告型问题还要展示 `generated_reports.evidence_json`。

现有前端智能问答页右侧“活动”面板已经展示 input/output。下一步可继续优化成更像时间线的证据卡片，并增加“一键复制工具证据 JSON”按钮。

## 8. 下一步优先级

P0：

- 用 `crawl_probe.py` 分别探测房天下、安居客、链家。
- 你提供可用 Header/Cookie 后，先只跑单页探测，不直接大规模入库。
- 选出 1 个主采源和 1 个补充源。

P1：

- 扩充房天下/安居客区县映射，尽量覆盖重庆全域。
- 优化大任务写库速度，必要时做批量 upsert。
- 前端采集任务页增加“新爬累计数量”和“旧库排除模式”。

P2：

- Agent 活动面板做成完整工具调用时间线。
- 给采集任务增加失败页一键重试。
- 为最终答辩导出新爬证据 JSON 和截图包。
